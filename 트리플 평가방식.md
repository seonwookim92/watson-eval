# 트리플 추출 평가 방식 (evaluate_triple.py)

## 개요

CTINexus Ground Truth와 모델이 추출한 트리플을 비교하여 두 가지를 평가한다.

1. **Triple Extraction** — 트리플(S, R, O)을 얼마나 정확하게 추출했는가
2. **Relation Type Matching** — 추출된 관계를 온톨로지 클래스로 얼마나 정확하게 분류했는가

---

## 1. Triple Extraction 평가

### 1-1. 두 가지 매칭 모드 (항상 동시 실행)

| 모드 | 비교 대상 | 트리플 표현 |
|---|---|---|
| **Soft (S+O)** | Subject + Object만 | `"CVE-2021-44228 [SEP] threat actor"` |
| **Full (S+R+O)** | Subject + Relation + Object 전체 | `"CVE-2021-44228 [SEP] exploited by [SEP] threat actor"` |

- Soft는 관계 표현 차이를 무시하고 개체 쌍이 맞는지만 본다.
- Full은 관계 표현까지 일치해야 매칭으로 인정한다.

### 1-2. 세 가지 매칭 방법

각 모드(Soft/Full)에 대해 아래 세 방법을 모두 적용한다.

#### Jaccard (토큰 집합 겹침)

```
similarity(a, b) = |tokens(a) ∩ tokens(b)| / |tokens(a) ∪ tokens(b)|
```

- 문자열을 단어 토큰 집합으로 변환 후 교집합/합집합 비율 계산
- Threshold ≥ 0.2 (기본값)이면 매칭으로 인정
- Greedy highest-score-first 방식으로 최적 쌍 구성
- 빠르고 결정론적이나 표현 차이에 취약 (하한선 역할)

#### Embedding (코사인 유사도)

```
similarity(a, b) = cos(embed(a), embed(b))
```

- `sentence-transformers/all-MiniLM-L6-v2`로 임베딩 후 코사인 유사도 계산
- Threshold ≥ 0.75 (기본값)이면 매칭으로 인정
- Greedy highest-score-first 방식으로 최적 쌍 구성
- 의미적 유사성 포착 가능 (약어, 패러프레이즈 처리)

#### LLM-as-a-Judge (의미 판단)

- LLM에게 Predicted 목록과 Gold 목록을 주고 의미적으로 동일한 쌍을 JSON으로 반환받음
- Soft 모드: `(Subject, Object)` 쌍을 보고 판단, 관계 차이 무시
- Full 모드: `(Subject, Relation, Object)` 전체를 보고 판단
- Soft/Full 각각 1회씩, 총 2회 LLM 호출 (asyncio.gather로 동시 실행)
- Gold 중복 매칭 방지 (먼저 나온 쌍 우선)
- 가장 정확하나 비결정론적 (핵심 기준)

### 1-3. 매칭 결과 → 지표 계산

6개 결과 (2 modes × 3 methods) 각각에 대해:

```
TP = 매칭된 쌍 수
FP = 예측했으나 GT에 없는 수  (= n_pred - TP)
FN = GT에 있으나 예측 못한 수 (= n_gold - TP)

Precision = TP / n_pred
Recall    = TP / n_gold
F1        = 2 × P × R / (P + R)
```

샘플 전체 집계:
- **Micro-F1**: 모든 샘플의 TP/FP/FN을 합산 후 계산 (큰 샘플에 가중치)
- **Macro-F1**: 샘플별 F1의 단순 평균 (샘플 크기 무관)

## 2. Relation Type Matching 평가

### 2-1. 설계 원칙

- **매칭 기반**: 트리플 매칭이 된 쌍에 대해서만 타입을 비교
- **LLM-Full 쌍 사용**: 가장 의미론적으로 정확한 LLM Full 매칭 결과를 기준으로 삼음
- **Eligible 쌍만 평가**: pred와 gold 양쪽 모두 relation type이 비어있지 않은 쌍만 대상

### 2-2. 타입 정규화

```python
"action:result"    → "result"         # 네임스페이스 접두사 제거
"createdBy"        → "createdby"      # 소문자 + 특수문자 제거
"https://.../uses" → "uses"           # URI에서 local name 추출
""  / "none" / "unknown" → 제외       # 빈 타입 마커 필터링
```

### 2-3. 계층적 부분점수 (OntologyHierarchy)

OWL 온톨로지의 `rdfs:subClassOf` 체인을 BFS로 탐색하여 두 타입 간 거리를 계산한다.

| 거리 | 점수 | 의미 |
|---|---|---|
| 0 (동일) | **1.0** | 정확히 일치 |
| 1 단계 | **0.6** | 부모 또는 자식 클래스 |
| 2 단계 | **0.3** | 조부모 또는 손자 클래스 |
| 3+ 단계 | **0.0** | 너무 멀거나 무관 |

```
예시 (UCO):
  pred: "ActionReferenceObject"
  gold: "ObservableObject"         ← pred의 1단계 부모
  → 점수: 0.6
```

양방향 탐색: pred → gold 방향과 gold → pred 방향 모두 확인.

### 2-4. 지표 계산

```
Score = Σ(hierarchical_score for each eligible pair)
Elig  = eligible 쌍 수 (양쪽 모두 non-empty type)
Acc   = Score / Elig
```

P = R = F1 = Acc (분모가 Elig로 symmetric하기 때문)

샘플 전체 집계:
- **Micro Acc**: Σ(Score) / Σ(Elig)
- **Macro Acc**: 샘플별 Acc의 단순 평균

## 3. 온톨로지별 Relation Type 커버리지

CTINexus GT의 relation type 주석 현황 (10개 파일 기준):

| 온톨로지 | 타입 있는 트리플 | 전체 | 커버리지 | exact | approximate |
|---|---|---|---|---|---|
| UCO | 6 | 124 | **4.8%** | 0 | 6 |
| STIX | 46 | 124 | **37.1%** | 23 | 23 |
| MALOnt | 31 | 124 | **25.0%** | 14 | 17 |

**해석**:
- UCO는 CTI 관계 동사(`discovered`, `patched`, `exploited`)가 UCO 액션 프로퍼티 체계와 맞지 않아 커버리지가 극히 낮음 (4.8%). Relation Type 평가 지표로 UCO는 신뢰하기 어려움.
- STIX는 `uses`, `targets`, `exploits` 등이 CTI 관계와 직접 대응되어 커버리지가 높음. **주요 Relation Type 평가 지표로 적합**.
- MALOnt는 악성코드 특화 관계에서 준수한 커버리지.
- UCO는 **엔티티 타입 평가**에 활용하고, Relation Type 평가는 **STIX/MALOnt 결과를 주지표**로 사용하는 것이 합리적.

## 4. 실행 방법

```bash
# 기본 실행 (Soft+Full 동시, explicit 트리플만)
python evaluate_triple.py \
    --results  outputs/watson-new_uco_results.json \
    --ground-truth datasets/ctinexus/annotation/

# implicit 트리플도 포함
python evaluate_triple.py ... --include-implicit

# 샘플 수 제한
python evaluate_triple.py ... --limit 5

# 결과 저장
python evaluate_triple.py ... --output outputs/eval_triple.json

# 임베딩/Jaccard threshold 조정
python evaluate_triple.py ... --emb-threshold 0.8 --jac-threshold 0.3
```

## 6. 설계 결정 근거

| 결정 | 이유 |
|---|---|
| Soft/Full 동시 실행 | 두 모드가 각각 다른 정보를 제공. 사용자가 필요에 따라 선택 |
| Jaccard = S+O or S+R+O (모드 따름) | Jaccard는 표현 차이에 취약해 Full에서 FN이 많아지지만, 엄격한 하한선으로서 의미 있음 |
| Type matching 기준 = LLM-Full | 의미론적으로 가장 정확한 매칭 쌍을 타입 비교 기준으로 사용 |
| Eligible만 평가 | 타입 없는 쌍을 분모에 포함하면 모델이 타입을 예측 안 하는 것이 유리해지는 왜곡 발생 |
| Gold 중복 제거 | 동일 gold가 여러 pred에 매칭되면 R > 1.0 발생하는 버그 방지 |
